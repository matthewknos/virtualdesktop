How the agent works

A Copilot Studio agent backed by Power Automate that reads a consolidated Workday-resident source of truth, drafts a cross-border comp package into a Teams adaptive card for Total Rewards, classifies the tax case against a versioned rule engine (routine cases only), and logs every recommendation to a 7-year equal-pay audit trail. Read-only across Workday and SharePoint in v1; the canonical Cross-Border Comp record is written in v2.

01 — Architecture

Architecture overview

Three input clusters feed the agent. The agent resolves context, freshness-checks every source, drafts a package, and fans out: a Teams adaptive card for Total Rewards, an equal-pay audit row for every recommendation, and (Bold v2 only) a canonical Cross-Border Comp record in Workday Extend.

02 — Trigger

Trigger model

Conversational, not event-driven. Two DM entry points; each one starts a single agent loop bound to one requisition.

  1. Hiring manager DM — "what does [name] cost in [city]?" The agent posts a one-screen preview to the hiring manager (no band inputs, no percentile language) and a full draft to Total Rewards for sign-off.
  2. Total Rewards DM — "package for [requisition id]". The agent posts the full draft straight to a Teams adaptive card for review.
Why conversational, not event-driven The trigger is a Talent Acquisition decision (open the conversation with the candidate), not a Workday state transition. Coupling the agent to a Workday business-process event would force the agent to run on every requisition opened — including ones that never cross a border. The DM model keeps the agent's surface area tight: it runs only when a human needs a package.

04 — Source of truth

The data lift is the project

Today the six inputs live in Workday (bands, benefits) plus six SharePoint sites (FX, COL, relocation, benefits parity, tax rules, legacy policy archive) with conflicting versions. Consolidation into Workday Extend is the precondition for everything else — ~6 weeks of integration work running in parallel to the agent build.

  • FX → XBC_FxRate, refreshed weekly via Power Automate from a licensed feed.
  • COL → XBC_ColIndex, refreshed quarterly from a named provider.
  • Relocation matrix → XBC_RelocationCorridor, keyed on (home, dest, level).
  • Benefits parity rules → XBC_BenefitsParityRule (Workday Benefits still owns the catalogues; this object owns the reconciliation logic).
  • Tax-equalisation rules → XBC_TaxRule (routine cases only; non-routine cases never enter this object).
  • Legacy policy archive → stays on SharePoint as a read-only reference link from each XBC_RelocationCorridor row.

Each object carries a source_version field stamped into every audit row so the quarterly fairness review can reproduce a recommendation against the exact input data that produced it.

Why Workday Extend, not a separate data platform The agent already reads Workday for bands, benefits, and HCM. Workday's audit retention and access-control model satisfies the 7-year equal-pay requirement out of the box. The Bold-v2 write surface (CrossBorderComp) lives in Workday Extend already — same governance plane, no parallel data estate to maintain.
05 — Draft pipeline

How a single draft gets built

Short-circuit pipeline. Every step logs inputs and outcome; the first failed freshness or missing-input check aborts and surfaces a refusal with the specific stale field. Only an all-pass run produces a Teams card.

1
Resolve subject, requisition, corridor Fail → refusal ("cannot resolve employee {id}" or "requisition {id} has no destination country"). logs: subject_id, requisition_id, home_country, dest_country, level, role
2
Freshness check across all six SoT objects Fail → refusal with the offending field ("FX rate GBP-EUR is 35 days old (limit 30)"). No partial draft. logs: fx_age, col_age, band_age, relocation_age, benefits_age, tax_rule_age
3
Read all inputs at clock-pin Snapshot source_version per field. The remainder of the draft is deterministic against this snapshot. logs: source_versions map
4
Classify tax case Routine → rule engine computes gross-up. Non-routine → flag_to_external_tax_advisor called; agent stops calculating, draft proceeds with referral ID and no tax position. logs: tax.class, tax.rule_id, tax.referral_id
5
Compute band position If position < 25 or position > 75 → justification_required = true; adaptive card flags the justification field as mandatory. logs: band_position_pct, justification_required
6
Draft the package Assemble base + relocation + benefits delta + tax + COL delta + FX-normalised view into the audience-appropriate adaptive card. logs: package payload
✓
Post + log Post adaptive card to the audience. Append one row to the equal-pay audit log with all inputs, source versions, decision provenance. logs: card_id, audit_row_id, posted_at
06 — Audit log

Equal-pay audit row schema

One row per recommendation — whether approved, edited, rejected, or referred to the external tax advisor. The schema is versioned because the quarterly fairness review (HRD + CFO + Legal) and any downstream Power BI dashboards depend on stable column names. 7-year retention.

07 — Guardrails

Guardrails (technical)

  • Freshness refusal. Any stale source → full refusal, no partial draft. Refusal copy names the field and its age vs. the cap. The agent does not produce a "best-effort" package when an input is past its limit.
  • Tax routine-only. Hard rule: the agent never computes a tax position outside the rule engine. Non-routine inputs to compute_routine_tax_equalisation raise; the agent must call flag_to_external_tax_advisor instead.
  • Confidentiality masking. Hiring-manager-audience messages are regex-scanned at post-time for band percentile labels, internal band IDs, source-version strings, "above band" / "below band". Any hit blocks the post.
  • Idempotency. Draft pipeline keyed by (requisition_id, attempt_id); a retry must not double-write to the audit log or double-post to Teams.
  • Justification gating. Approval transitions are rejected at the tool boundary when justification_required is true and justification is empty. Cannot be bypassed by the agent or the reviewer.
  • Audit retention. Every row carries retention_until = timestamp + 7 years, enforced at the storage layer. Justification text is sealed on approval; subsequent edits create a new row, not an in-place mutation.
  • Kill switch. Total Rewards lead can disable draft posting globally via a config flag in minutes; the agent reverts to refusal-with-message on every request.
  • Schema versioning. Audit log and (Bold v2) CrossBorderComp schemas are semver-versioned. Breaking changes require a coordinated bump — the quarterly fairness review consumes these schemas.
08 — Evaluation

Evaluation harness

Replayable eval that gates ship of v1. 24 golden cases (12 UK↔IE + 12 UK↔PL) cover routine in-band drafts, justification-mandatory above/below-band, CFO-routing relocation, short-term assignment, and internal transfer. 12 adversarial cases cover every named guardrail: stale FX / COL / band / tax rule, missing band, equity-vesting non-routine, dual-citizenship non-routine, out-of-scope corridor, repatriation with band drift, collapsed-band pay edge, and a confidentiality-breach attempt.

Run in a Workday sandbox with clock pinning so freshness is deterministic. Pre-pilot gate (week 9): ≥22/24 golden, 12/12 adversarial, zero legal-flagged equal-pay issues. Re-runs on every change to agent prompt, tool config, or any XBC_* source row.

09 — Build phases

Build phases

  • Upstream — do regardless. ~6 weeks. Consolidate the six inputs into Workday Extend (XBC_* objects). Refresh jobs, schema sign-off, Data Owner per object. The agent is meaningless without this; the consolidation is independently valuable even if v1 is descoped.
  • Pragmatic v1. 10 weeks. UK↔IE + UK↔PL drafter. Read-only across Workday and SharePoint. Posts adaptive card; logs audit row. Total Rewards reviews, edits, approves, or refers. No write to CrossBorderComp.
  • Bold v2. Adds the CrossBorderComp write surface and state machine, full jurisdictional coverage (UK↔DE, UK↔US, UK↔CA, IE↔PL, intra-EU), Workday Talent Mobility pairing, and the candidate-side comp transparency view required by the EU Pay Transparency Directive.

Tool contracts, conversation transcripts, the full CrossBorderComp schema, and the source-of-truth migration plan live in the build briefs (02-bold-build-brief.md, 02-pragmatic-build-brief.md) and the schema doc (03-source-of-truth-schema.md) in the repository.
