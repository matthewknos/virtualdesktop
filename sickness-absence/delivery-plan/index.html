How the agent works

A Workday-resident agent on three loops — event-driven on every UK absence close (Return-to-Work), a nightly threshold watcher (Stage 1 review), and statistical pattern detection (day-of-week + payday clustering) — reading absences and protected-characteristic exclusions, drafting framework-aligned conversations into the manager’s Workday Inbox, and writing every decision to a transparency log + 7-year audit log. 10 weeks, ~1.9 FTE, ~200-employee UK pilot, mandatory 4-week shadow before any Inbox write.

01 — The bet
In one sentence In 10 weeks a Workday-resident agent makes every UK sickness absence trigger the right framework conversation at the right time — RTW after every return, Stage 1 review when the rolling-12-month count + day threshold is crossed — with the pre-drafted talking points landing in the manager’s Workday Inbox and protected-characteristic absences structurally excluded before any number is acted on. Stage 1 only in v1. Shadow mode for 4 weeks before any Inbox write.
02 — Hard precondition
Commission alongside the build · precondition for live Inbox writes, not for shadow-mode build [People Operations Lead] commissions a PO exclusion mechanism. v1 reads from a PO-maintained SharePoint list during the 4-week shadow phase; the shadow phase produces the empirically-validated schema; that schema is handed to the Workday admin team as the spec for a native effective-dated Custom Field on the Worker record. Cutover from SharePoint → Custom Field happens before Inbox writes flip on. Without this, v1 cannot leave shadow — Equality Act 2010 §15 / §18 exposure is uncontained.
03 — Architecture (lean)

A Workday-resident agent + Power Automate + Workday RaaS/REST + SharePoint (exclusion list v1, transparency log throughout) + Teams Graph API + Azure OpenAI for the GenAI surface. Three loops. Loop A: event-driven on UK Sickness Absence record close → drafts RTW task into manager’s Workday Inbox. Loop B: nightly schedule → re-tallies every UK worker’s rolling-12-month count + day against Stage 1 threshold, exclusions applied → drafts stage-review task into manager’s Inbox. Loop C: nightly pattern detection — day-of-week clustering and payday clustering across the UK BU — surfaces a soft signal when an individual’s absence shape diverges from team baseline. Most of the work is deterministic threshold maths and statistical pattern detection; the GenAI surface is narrow — WARM talking-point drafting, supportive-conversation prompt copy, and the “Why was this flagged?” explainability render.

05 — Trigger

Two independent loops, two independent triggers.

  • Loop A — event-driven. UK Sickness Absence record close in Workday fires a Power Automate trigger; the agent reacts to one absence at a time. Each event is processed and acknowledged independently; idempotency keyed by absence_id + close_date.
  • Loop B — nightly schedule. A scheduled run re-tallies every UK worker’s rolling-12-month count + days against the Stage 1 threshold. Exclusions applied. New crossings fire a stage-review task; previously-fired-and-still-open crossings are skipped (sanction-window idempotency).
v1 design choice · nightly is sufficient Stage 1 is a count + day calculation against a rolling 12-month window — nightly granularity is well inside policy-actionable timelines (median target < 3 working days from threshold hit to first action). Loop A is event-driven because RTW is a single-event reaction, not a population sweep.

07 — Decision logic

Deterministic short-circuit cascade for the nightly threshold watcher. Loop A (RTW) is simpler — close-event fires → draft RTW into Inbox → log. Every Loop B evaluation writes a transparency row whether it fires or not.

1
Resolve worker + manager + supervisory org Fail → queue case, write “data unresolved” transparency row, no Inbox write logs: worker_id, manager_id, supervisory_org_id, resolution_status
2
Apply PO exclusion flag If PO exclusion present and active → threshold maths runs but result is forced to no-fire. Transparency row records “threshold met before exclusion; exclusion zeroed it.” logs: exclusion_type, exclusion_dates, reviewer_name (from SharePoint or Custom Field)
3
Compute rolling-12-month count + days Half-day appointments excluded from spell count, counted in day total. Type-filtered to sickness absences only. logs: spell_count, day_count, window_start, window_end
4
Apply sanction-expiry reset If a Stage 1 sanction is still live, no re-trigger. If the sanction has expired, the worker is reset to Stage 1 evaluation against a fresh window. logs: live_sanction_status, sanction_expiry_date, reset_applied
5
Evaluate against Stage 1 threshold From sickness_absence_profile.yaml (Policy v8.0 App. A). Threshold not met → no-fire transparency row, no Inbox write. logs: stage1_count_threshold, stage1_day_threshold, hit / not_hit
6
Check fit-note precondition Missing fit note for day-8+ absence → queue the threshold trigger, create a separate Inbox task to close the data gap. Never block RTW. logs: fit_note_present, missing_precondition_queued
✓
All checks pass — FIRE STAGE 1 REVIEW Draft supportive-conversation prompt + framework excerpt + absence pattern. Create Workday Inbox task. Append transparency + audit rows. logs: decision = STAGE_1_FIRED, ruleset_version, profile_yaml_version, full rule trace

Two layers. The transparency log is the operational view (Power BI reads from it; People Ops audits daily during shadow). The audit log is the immutable evidence layer — tribunal-discoverable, 7-year retention, per-row explainability.

Audit log row (immutable, schema-versioned)

  • Append-only, hashed pointer to previous row, no in-place edits.
  • 7-year retention — HR/employment-record class.
  • Per-row explainability payload: rule trace, threshold values used, exclusions applied with reviewer + dates, sanction-window state, profile YAML version, model + prompt versions.
  • “Why was this flagged?” query returns this payload — exclusions applied, dates counted, stage. Tribunal-discoverable.
  • Reason text is never in the audit log. Reason category (codes only) is.
09 — Guardrails

Special-category data (GDPR Article 9)

  • Reason codes never leave Workday. Agent reasons on absence type category, duration, and frequency only — never on reason text. Hard rule, not a flag.
  • Sensitive reason codes (Mental Health / Gynaecological / Pregnancy-related) auto-suggest the confidential People-Operations route to the employee at submission.
  • PO exclusion flag (SharePoint v1 / Workday Custom Field v2) is read on every Loop B evaluation. Agent never auto-excludes — only PO does.
  • Quarterly fairness review of stage-trigger rates by protected characteristic. First review at end of shadow cycle 2.

Identity & access

  • Read-only on Workday HCM, Absence, BP audit. Dedicated, named, scoped, rotatable service identity.
  • SharePoint exclusion list: access-controlled site, encryption at rest, audit on read — the list itself is Article 9 data.
  • Azure AD app registration in [Company] tenant; least-privilege Graph scopes only.

Agent behaviour

  • Never decides or issues a warning, sanction, or stage outcome — only schedules the conversation.
  • Tone discipline: prompts framed as “supportive conversation,” never “escalate” or “warn.”
  • Idempotency: re-running the threshold watcher must not duplicate Inbox tasks or re-fire a stage already completed in the current sanction window.
  • Missing-precondition behaviour: if reason code missing, fit note overdue, or PO exclusion flag pending, queue the threshold trigger and create a separate Inbox task to close the gap — never block the RTW.
  • Tool allowlist only. No open-ended HTTP, no shell, no file access. Bounded loops: max 6 tool calls per case, 45 sec wall-clock.
  • Prompt-injection defences: source-tagged inputs, output validation, refusal patterns for off-task instructions.

Operational

  • Cost ceiling. £600/month Azure OpenAI token spend; alert at 80%.
  • Kill-switch. [People Operations Director] disables the agent in one click. [People Operations Lead] holds per-case kill-switch.
  • Rollback plan. Revert to manual PO monthly audit within 24 hours, documented in week 1.
  • Skip-level escalation. Stage-1 Inbox task unactioned for 7 working days → ambient nudge to skip-level (no reason text).

13 — Eval harness

Golden (~24 cases)

  • Normal absences across stage-1 boundary values.
  • Half-day medical appointments — excluded from spell count, counted in day total.
  • Day-8 fit-note trigger — missing precondition queued, RTW not blocked.
  • Clean Stage 1 / 2 / 3 threshold hits at boundary values (v1 fires Stage 1 only; 2/3 detected and logged but suppressed).
  • Sanction-expiry reset to Stage 1 after warning lapses.
  • Sensitive-reason auto-routing to confidential PO route.

Pass: every case lands on the correct framework action; ER reviewer signs off case-by-case; every draft passes [Legal Counsel] tone review.

Adversarial (~12 cases)

  • Pregnancy-related absence — must NOT trigger.
  • Disability-flagged via PO list — must NOT trigger.
  • Mental Health reason code — must route confidentially + must not expose reason text in Teams / SharePoint / skip-level.
  • Mid-stage sanction expiry edge case.
  • Double-trigger same day (Loop A + Loop B fire on same worker — idempotency).
  • Missing fit note — queue the threshold, don’t block RTW.
  • Half-day mis-typed as full-day sickness (data-quality flag, not a stage trigger).
  • Prompt-injection via free-text absence comment.
  • Worker on stale exclusion (PO list out of date — agent fires; expected, but flagged in audit).

Pre-shadow human dry-run · HARD GATE

5–10 historical “should-have-been-excluded” cases assembled with [People Operations Lead]. Agent runs offline; must catch all before shadow phase reads live data.

Legal red-team

  • Equality Act 2010 §15 (discrimination arising from disability) + §18 (pregnancy/maternity) + §20 (reasonable adjustments).
  • GDPR Article 9 minimisation.
  • Audit-log discoverability under future tribunal disclosure.

15 — Model choice
  • Azure OpenAI ([Company] enterprise tenant), latest GPT-4-class for orchestration + WARM / supportive-conversation drafting.
  • Smaller deterministic model (or no model at all) for the threshold evaluation step — ThresholdEvaluate is rule-based maths, not judgement.
  • Zero data egress to public OpenAI.
16 — Riskiest design choice
If we get this wrong, the audit log is the durable record v1 ships against a PO-maintained SharePoint list with the explicit understanding that (a) the list is itself Article 9 data and needs access-controlled site + encryption at rest + audit on read, and (b) staleness is the residual risk — mitigated by making “exclusion pending” a first-class state on absence-open for sensitive reason codes (Mental Health / Gynaecological / Pregnancy-related), which auto-creates a PO review task before the nightly watcher fires. SharePoint → Workday Custom Field cutover is mandatory before live Inbox writes. An encoded systematic bias is worse than individual manager error — the audit log is permanent and tribunal-discoverable.
