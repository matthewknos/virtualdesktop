Sickness Absence Framework Coach

A Workday-resident agent that turns the UK Sickness Absence Framework from “manager remembers to do it” into “Workday Inbox makes sure it happens — with the conversation pre-drafted and protected absences structurally excluded.”

01 — Headline messages

The UK Sickness Absence Framework launched 1 April 2026. Two months in, the dominant failure per [Global People Ops Lead] is the same one Stage 2/3 always exposes: managers reach the threshold conversation having missed the earlier steps.

The loss

Every UK absence creates two unprompted obligations — a Return-to-Work conversation when the worker comes back, and a stage-review check when count + day thresholds are crossed in the rolling 12 months. Today both rely on the manager remembering. RTW conversations get skipped or logged via ad-hoc email. Stage prompts are caught by a monthly People Ops audit — often after Stage 2/3 has already arrived. The framework is “very clunky” per the [Global People Ops Lead], and the cost is consistent application across ~200+ episodes a month.

The signal

Workday already records every absence with type, dates, fit-note status, manager, supervisory org. The Workday Inbox is where managers already do BP work. The framework’s stage thresholds are mechanical — count + days over a rolling window — and explicit in Policy v8.0 Appendix A. We’re not inventing the maths or a new tool surface; we’re wiring an existing dataset to an existing inbox with a draft conversation pre-loaded. What Workday cannot do is detect the shape of absences — day-of-week clustering, payday clustering. Loop C adds that as a soft signal alongside the threshold maths.

03 — Problem
“Very clunky. People reach Stage 2 or 3 having missed the earlier prompts.” — [Global People Ops Lead]

Managers have a job aid and a SharePoint policy. RTW depends on the manager noticing the return. Stage prompts depend on the manager — or the monthly People Ops audit — catching the threshold. Two months after launch the dominant failure mode is reaching Stage 2/3 having had no Stage 1 conversation. And every miss on a protected-characteristic absence is an Equality Act 2010 exposure no audit trail can defend.

04 — Solution
  1. Loop A — event-driven RTW. A UK Sickness Absence record closes → agent drafts an RTW task into the manager’s Workday Inbox with WARM-framed talking points. Sensitive reasons (Mental Health / Pregnancy-related / Gynaecological) surface a confidential People-Ops route to the employee; reason text never leaves Workday.
  2. Loop B — nightly threshold watcher. Re-tallies every UK worker’s rolling-12-month count + day against Stage 1. Half-day appointments excluded from spell count. PO-flagged absences excluded entirely. When a threshold crosses (after exclusions), drops a stage-review task into the manager’s Inbox with framework excerpt + absence pattern + draft prompt.
  3. Loop C — nightly pattern detection. Runs day-of-week clustering and payday clustering across the UK BU. Surfaces a soft signal (not a stage record) when an individual’s absence shape diverges from team baseline at p<0.05, minimum n=4, ratio ≥3× baseline. Bradford-style scoring is locked out of stage logic; Loop C produces a separate signal in its own lane — dismissible without paperwork.
  4. Tone discipline. Drafts say “schedule a supportive conversation,” never “escalate” or “warn.”
  5. Transparency log. Every decision (fire or no-fire) writes a row to SharePoint; Power BI on top gives People Partners the population view.
  6. Never decides outcomes. Never auto-excludes. Never exposes reason text outside Workday. The agent prompts the conversation; the human has it.

06 — Two options on the table
v1Pragmatic — “Sickness Absence Framework Coach”

Loop A end-to-end + Loop B Stage 1 + Loop C day-of-week + payday clustering. One UK BU, ~200 employees. PO exclusion list lives in SharePoint for v1; cuts over to a native Workday Custom Field before live writes flip on. 4-week shadow phase — agent writes only to the transparency log; Inbox + Teams suppressed — before any manager sees a task. Pattern signals also run in shadow with weekly People-Ops FP review.

v2Bold — “UK Absence Stage 2/3 + Cross-Jurisdiction”

Same agent shape, broader signal. Stage 2/3 detection turned on, Teams ambient nudges live, cross-jurisdiction (IE framework on top of UK), advanced pattern detection (peri-deadline, post-leave returner). Gated on v1 hitting shadow-phase pass criteria.

Upstream fix (do regardless)

Build the PO exclusion mechanism. Workday today has no field or PO-managed list to mark an absence as discounted from threshold counts. Without it the agent could trigger Stage 1 on disability- or pregnancy-related absences — Equality Act 2010 §15 + §18 exposure, durably recorded in the 7-year audit log. v1 reads from a PO-maintained SharePoint list during the shadow phase; the schema empirically validated during shadow is handed to [Workday Admin Team Lead] as the spec for a native effective-dated Custom Field on the Worker record. Cutover from SharePoint → Custom Field is mandatory before live Inbox writes. Owner: [People Operations Lead] + Workday admin team.

07 — Adoption / change asks

The build is small; the discipline around it is the lift. None of this is optional.

  • PO maintains the exclusion list. Effective-dated rows per worker with exclusion type (disability / pregnancy / gynae-discretionary / mental-health-discretionary), reviewer name, dates. The list itself is Article 9 data — access-controlled site, encryption at rest, audit on read.
  • Sensitive-reason auto-routing. Mental Health / Pregnancy-related / Gynaecological reason codes auto-suggest the confidential People-Ops route to the employee at absence submission — closing the window between absence-open and PO triage.
  • WARM template sign-off. [Legal Counsel] + ER reviewer sign off every coaching template before pilot. Versioned, 2-eyes review on changes.
  • Kill-switch ownership. [People Operations Director] holds the agent-wide flag. [People Operations Lead] holds per-case kill-switch. Drill week 9.
  • Quarterly fairness review. Stage-trigger rates by protected characteristic, run by [People Operations Lead] + AI CoE Sec. First review at end of shadow cycle 2.
  • Audit log retention. Immutable, append-only, schema-versioned, 7-year retention (HR/employment-record class), per-row explainability (“Why was this flagged?” → exclusions applied, dates counted, stage).

09 — Workday overlap

BUILD, wrapping existing Workday business processes. No Workday agent in portfolio detects the count + day cadence, and Workday cannot configure pattern triggers (day-of-week or payday concentration) at all; the integrations we touch are read-only on the existing surface.

  • Workday Self-Service Agent answers policy questions — it does not detect thresholds or draft conversations. Adjacent, not competing.
  • Workday Case Agent handles generic case routing. We invoke it indirectly via the existing BP for Stage 2 People-Ops routing; we don’t replace it.
  • Workday BP Optimize / Inbox AI — when GA, may surface Inbox prioritisation. Our Inbox task slots into that surface; we’re additive.
  • This agent owns the threshold-detection + draft action; exceptions and Stage 2/3 escalation inherit the existing Workday BP routing — we invent no new reviewer flow.
10 — Risks

Conservative defaults at launch. People Ops keeps oversight; the agent never owns the framework.

RiskMitigation
RiskAbsence-data quality. Mis-categorised reason codes, late or missing fit notes, half-days mis-typed as full-day sickness. Signal: shadow-mode audit >10% mismatch with manual PO count. MitigationSprint 1 pulls 6 months of UK absence data; run Loop B offline against known live sanctions before the shadow phase touches live data. Missing-precondition behaviour queues threshold triggers when fit notes overdue; never blocks RTW.
RiskDiscrimination exposure. Without a robust PO exclusion mechanism, the agent could trigger Stage 1 on disability- or pregnancy-related absences — Equality Act 2010 §15 / §18. The audit log is permanent and tribunal-discoverable; an encoded systematic bias is worse than individual manager error. MitigationSharePoint exclusion list during shadow; Workday Custom Field before live writes; pre-shadow human dry-run on 5–10 historical “should-have-been-excluded” cases (must catch all). Quarterly fairness review of stage-trigger rates by protected characteristic.
RiskManager non-adoption. The probation prototype surfaced that managers screen out Workday Inbox tasks. Signal: <50% of RTW tasks actioned within 5 days in shadow mode. Mitigation2-week ride-along with 3 managers — observe whether they action existing Inbox tasks before adding more. Skip-level ambient nudge if Stage-1 task unactioned for 7 working days (no reason text carried).
RiskPattern false-positives. Loop C runs statistical detection on small-n absence data — Friday/Monday clustering and payday concentration. Higher false-positive risk than threshold counting; surfaces could feel intrusive if dismissed weekly without review. MitigationHard fire criteria — min n=4 spells, ratio ≥3× team baseline, binomial p<0.05. Dismissed-with-reason events reviewed weekly by People Ops during shadow. Loop C signals are explicitly soft — dismissible without paperwork, never written as a stage record. Bradford scoring locked out of stage logic by design.
RiskBradford retrofit risk. Pressure to extend Loop C into return-to-work / short-spell clustering would creep Bradford-style scoring into stage logic — exactly what v2 (Bradford-as-context, not as logic) ruled out. The framework’s count + day maths is the audit anchor; conflating patterns with stages weakens that. MitigationRTW short-spell clustering deferred to v2 module review only — never to v1. Loop C ships only the two least-Bradford-adjacent patterns (day-of-week, payday). Quarterly fairness review includes a check that Loop C signals do not correlate with stage triggers more than chance.
RiskPolicy source not anchored in repo. Threshold values (count, days, rolling window, half-day exclusion) currently live in brief authors’ memory; the Policy v8.0 PDF is not checked into the repo. Config-externalisation without a checked-in source is a brittle audit position. MitigationSprint 1 sign-off gate: check the policy into policy-sources/ or formally pin the version in the sickness_absence_profile.yaml header. Either form is fine; both is best.
11 — Recommendation

Ship the Pragmatic option as the v1 PoC: Loop A end-to-end, Loop B Stage 1, Loop C day-of-week + payday clustering, SharePoint exclusion list during shadow, 4-week shadow mode before any Inbox write, SharePoint → Workday Custom Field cutover before live writes flip on.

If shadow phase hits ≥95% threshold-calc agreement with manual PO audit and zero false triggers on PO-flagged exclusions, v2 (Stage 2/3 + cross-jurisdiction + advanced patterns) is funded. Stage 2/3 detection, Teams ambient nudges live, cross-jurisdiction (IE), and advanced pattern detection (peri-deadline, post-leave returner, multi-pattern stacking) follows on the same Loop C surface. Bold-first here would over-build advanced pattern detection and Stage 2/3 logic before the agent and the human PO process are calibrating against the same exclusion list.

12 — Build approach
MVP — weeks 1–8 (shadow)
Loop A + Loop B Stage 1 + Loop C (day-of-week + payday clustering) against seeded then live data. SharePoint exclusion list. Transparency log + audit log. Inbox + Teams writes suppressed. Pre-shadow dry-run on historical excluded cases (hard gate). Pattern signals also run in shadow with weekly People-Ops false-positive review.
Phase 2 — weeks 9–10 (cutover + live)
SharePoint → Workday Custom Field migration. Inbox writes flip on; Loops A, B, and C go live in parallel. Teams ambient nudges enabled. Manager ride-along + adoption monitoring. Decision gate on v2 build.
Phase 3 — v2 (gated)
Stage 2/3 detection turned on. Cross-jurisdiction (IE framework on top of UK). Advanced pattern detection on Loop C surface — peri-deadline, post-leave returner, multi-pattern stacking. Same agent shape; broader signal surface for People Partners.
