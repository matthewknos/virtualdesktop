R&R Agentic Approver

An agent that triages every R&R nomination — auto-clearing the ~95% that fall inside policy, and handing the ~5% that need human judgment to People Ops with the reasoning already attached.

01 — Headline messages

The person on the ground said it: “200 sitting with the team to approve… 95% of the time, very much just press a button.”

The loss

Every R&R cycle, ~200 nominations queue in a People Ops inbox waiting for a click. ~190 of them are pure rubber-stamp — value inside policy, recipient eligible, frequency clean. The team burns ~2 days per cycle scanning and approving. With ~6 cycles a year, that’s ~12 People Ops days/year doing work that requires no judgement. Meanwhile recognition that should land the same hour drifts into days, and the manager who nominated forgets they ever did.

The signal

The policy is already fully expressible as a rule cascade — value cap, eligibility, frequency. Workday already holds the worker, manager, position and 90-day history data the cascade needs. The R&R business process already exists and routes to the right reviewer when something looks off. We’re not inventing routing; we’re auto-clearing the in-policy ~95% so the human reviewer only sees the ~10 cases that need judgement.

03 — Problem
“200 sitting with the team to approve… 95% of the time, very much just press a button.”

Of every 200 R&R approvals each cycle, ~190 are pure rubber-stamp — open inbox, scan line, click approve. The control exists, but the human judgment inside it is rare. People Ops spends ~2 days per cycle on work that, by their own account, requires no analysis.

04 — Solution
  1. Reads each new R&R nomination at submission and resolves recipient, nominator and project context from Workday.
  2. Decides — auto-approves when policy checks come back clean (value, frequency, recipient eligibility, nominator legitimacy); otherwise routes for human review with a tagged reason.
  3. Hands each exception to the existing Workday BP reviewer with the specific reason it was flagged and the context the reviewer would otherwise have to look up.
  4. Watches for patterns the rule cascade alone can’t see — reciprocity, ring nominations, off-policy bursts (Phase 2).
  5. Writes every decision, with the agent’s reasoning, to a SharePoint Excel sheet / Workday worksheet for full team transparency.

06 — Two options on the table
v1Pragmatic — “R&R Rubber-Stamp Auto-Approver”

3-rule cascade, shadow mode cycle 1, then live auto-approve on the in-policy ~95%. Exceptions inherit existing Workday BP routing with a tagged reason. SharePoint audit row per decision.

v2Bold — “Workday Approval Audit Agent”

Generalise the pattern across multiple rubber-stamp Workday BPs in People Ops (R&R first, then sickness, then probation, then ad-hoc letters), with pattern-detection (reciprocity, ring nominations, off-policy bursts) baked in.

Upstream fix (do regardless)

Version the R&R policy. Today it lives partly in a SharePoint page, partly in tribal knowledge — the value cap, the eligibility carve-outs, the frequency rule. Pull it into a single versioned ruleset (ruleset_v1.0.yaml-style) that the agent reads at decision time. Without this, the agent and the human approver are calibrating against different mental models, and shadow-mode comparison is meaningless. Owner: [People Ops Lead] + R&R policy owner. ~2 weeks.

07 — Adoption / change asks

The build is small; the discipline around it is the lift. None of this is optional.

  • Versioned policy. Thresholds (value cap, eligibility carve-outs, frequency window) re-validated and bumped each refresh cycle. Every decision row records ruleset_version.
  • Weekly audit owner. [People Ops Lead] nominates the reviewer who eyeballs the 5% SharePoint sample post-launch. Drift detection, not gatekeeping.
  • Exception-reason taxonomy training. 10-min walkthrough for BP reviewers on the five tags — value, eligibility, frequency, data-unresolved, pattern — so the routed ~10/cycle land with a reason the human can act on without triage.
  • Kill-switch ownership. [People Ops Lead] holds the config flag. One flip reverts the agent to shadow mode across the board in <5 min. Drill it in week 9.

09 — Workday overlap

BUILD, on top of an existing Workday BP. No Workday portfolio overlap blocks this.

  • Workday Self-Service Agent handles policy Q&A only — it does not approve anything. No conflict.
  • Workday Approval Audit Agent (future generalisation, ROADMAP row #53) is the v2 path — auditing rubber-stamp approvals systemically across BPs.
  • This agent owns the auto-approve action on R&R nominations; exceptions inherit the existing Workday BP routing — we invent no new reviewer flow.
10 — Risks

Conservative defaults at launch. People Ops keeps oversight; the agent never owns the policy.

11 — Recommendation

Ship the Pragmatic option as the v1 PoC: 3-rule cascade (value, eligibility, frequency), exception routing inheriting the existing BP with a tagged reason, SharePoint audit row per decision, shadow-mode cycle 1 as the hard gate.

If shadow-mode cycle 1 lands ≥95% agreement with People Ops on in-policy nominations and zero false-approves on exceptions, v2 (Bold — Workday Approval Audit Agent, Phase 2 pattern-detection layer baked in, generalisation across other rubber-stamp Workday BPs) is funded. Bold-first here would over-build before we know the agent and the human are reading the same policy.

12 — Build approach
MVP
Policy-threshold auto-approve, exception routing via the existing Workday BP, SharePoint audit sheet / Workday worksheet.
Phase 2
Pattern-detection layer — the agent reads cross-nomination signals (reciprocity, frequency anomalies, off-policy values, gaming) the rule cascade alone can’t see.
Phase 3
Generalise the agent pattern across other rubber-stamp Workday business processes.
