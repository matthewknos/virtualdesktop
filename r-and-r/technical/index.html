How would we build it

An event-driven agent that reads R&R nominations from Workday, applies versioned policy thresholds, completes the BP step for in-policy nominations, and writes every decision to a SharePoint Excel sheet or Workday worksheet via Microsoft Graph (or Workday RaaS write-back).

01 — Architecture

Architecture overview

Three Workday surfaces feed the agent core. The agent applies policy and fans out to two destinations: Workday for the BP write, SharePoint for the audit row. Exceptions are not re-routed — they fall back to the BP's existing reviewer routing.

02 — Trigger

Trigger model

Event-driven on R&R nomination submission in Workday. The agent reacts to a single nomination at a time; each event is processed and acknowledged independently.

TBD · Primary integration unknown Confirm whether Workday R&R exposes a nomination-submitted event hook (webhook, Workday Studio outbound, or business-process subscription). If no event surface exists, the fallback is a short-interval RaaS poll (e.g. every 60s) against a custom report filtered to nominations submitted since last cursor. The poll path is acceptable for MVP — event-driven is preferred for Phase 2.
03 — Integration

Integration surface

Five touchpoints. Four read, two write (R&R module reads and writes; SharePoint is write-only from the agent). The agent invents no new routing — exceptions inherit the BP's existing reviewer flow.

04 — Decision logic

Decision logic

A short-circuit rule cascade. Any failure routes to exception with a tagged reason; only an all-pass result triggers an auto-approval. Every step is logged in the SharePoint row regardless of outcome.

1
Resolve recipient and nominator records Fail → route as exception ("data unresolved") logs: recipient_id, nominator_id, resolution_status
2
Check value cap Fail → route as exception ("value above auto-approve cap") logs: nomination_value, cap_threshold, pass/fail
3
Check recipient eligibility Active employee, not on a formal performance process. Fail → route as exception ("eligibility") logs: active_status, performance_process_flag, pass/fail
4
Check frequency rule Per recipient / nominator / project, 30-day window. Fail → route as exception ("frequency") logs: window_count, threshold, pass/fail
5
Check manager-self-loop pattern (Phase 2) Detect manager nominating direct reports who reciprocally nominate the manager. Fail → route as exception ("pattern") logs: pattern_signal, pattern_score, pass/fail
✓
All checks pass — AUTO-APPROVE Complete BP step in Workday and append row to SharePoint. logs: decision = AUTO-APPROVED, ruleset_version, full rule trace

06 — Guardrails

Guardrails (technical)

  • Idempotency. Decisions keyed by nomination_id; a re-fired event must not double-write to the BP or duplicate audit rows.
  • Hard cap. Auto-approval is refused above a fixed monetary value regardless of rule logic — an unconditional ceiling sitting outside the rule cascade.
  • Reversibility. Auto-approvals are standard BP completions — reversible by People Ops in one Workday action. No exotic state to unwind.
  • Kill switch. People Ops lead can disable auto-approval globally via a config flag in minutes; the agent reverts to logging-only (decision_mode = shadow).
  • Audit. Every decision is appended as a SharePoint row, including the failed checks for exceptions — full transparency, not just the approvals.
  • Output contract. SharePoint row schema is semver-versioned. Downstream Power BI dashboards depend on it; breaking changes require a coordinated bump.
07 — Build phases

Build phases

  • MVP. 3-rule threshold check (value cap, eligibility, frequency) plus exception routing via the existing BP and SharePoint append. One R&R category, one cycle.
  • Shadow mode (cycle 1). decision_mode = "shadow" — rows logged only, no BP writes. Compare agent decisions against human decisions on the same nominations to calibrate before going live.
  • Phase 2. Abuse-pattern detection (manager-self-loop, gaming patterns), multi-category rollout.
  • Phase 3. Generalise into Agent #53 (Workday Approval Audit Agent) across other rubber-stamp BPs.
