How the agent works

An event-driven agent that reads each R&R nomination from Workday, resolves the recipient and project context, applies versioned policy checks, decides — auto-approving when checks are clean and handing exceptions to existing BP reviewers — and writes every decision with its reasoning to a SharePoint Excel sheet / Workday worksheet via Microsoft Graph (or Workday RaaS write-back).

01 — The bet
In one sentence In 10 weeks, an event-driven agent (or RaaS-polled fallback) reads each R&R nomination at submission, runs a 3-rule policy cascade (value cap, eligibility, frequency) against versioned policy, auto-approves the clean ~190/200, routes the ~10 exceptions to the existing Workday BP reviewer with a tagged reason, and writes every decision to a SharePoint Excel sheet / Workday worksheet — taking People Ops time per cycle from ~2 days to <30 min and nomination-to-recognition from days to the same hour.
02 — Hard precondition
Week-1 decision · not blocking for v1 build, but the precondition for v2 [People Ops Lead] + Workday IT lead sign off the trigger model in week 1: either Workday R&R exposes a nomination-submitted event hook (webhook / Studio outbound / BP subscription) — preferred — or the agent runs a 60-second RaaS poll against a custom report filtered to nominations submitted since last cursor. v1 ships against whichever surface exists at week 5; the event-hook path is the precondition for v2 pattern-detection at scale. Without a decision in week 1, week 3 build slips.
03 — Architecture (lean)

Copilot Studio agent + Power Automate trigger (Workday R&R event hook if available, else 60s RaaS poll against a custom report) + Workday HCM REST (worker / manager / position reads) + Workday RaaS (90-day nomination history per recipient / nominator / project) + SharePoint via Microsoft Graph (decision-row append) + Workday R&R BP completion call (auto-approve only). Two paths through the agent: (1) all checks pass → complete BP step + append SharePoint row tagged AUTO-APPROVED; (2) any check fails → leave BP for existing reviewer with comment + append SharePoint row tagged ROUTED FOR REVIEW with reason. Shadow mode = path (1) downgraded to log-only.

07 — Decision logic

A short-circuit policy cascade. Each check is logged with its outcome and the reasoning that produced it; the first failure routes the nomination to a human reviewer with a tagged reason; only an all-pass result triggers an auto-approval.

1
Resolve recipient and nominator records Fail → route as exception (“data unresolved”) logs: recipient_id, nominator_id, resolution_status
2
Check value cap Fail → route as exception (“value above auto-approve cap”) logs: nomination_value, cap_threshold, pass/fail
3
Check recipient eligibility Active employee, not on a formal performance process. Fail → route as exception (“eligibility”) logs: active_status, performance_process_flag, pass/fail
4
Check frequency rule Per recipient / nominator / project, 30-day window. Fail → route as exception (“frequency”) logs: window_count, threshold, pass/fail
5
Check manager-self-loop pattern (Phase 2) Detect manager nominating direct reports who reciprocally nominate the manager. Fail → route as exception (“pattern”) logs: pattern_signal, pattern_score, pass/fail
✓
All checks pass — AUTO-APPROVE Complete BP step in Workday and append row to SharePoint. logs: decision = AUTO-APPROVED, ruleset_version, full rule trace

09 — Guardrails

Identity & access

  • Read-only on Workday HCM, RaaS, and the R&R nomination payload. Write only to the Workday R&R BP completion endpoint and the SharePoint audit list.
  • Azure AD app registration in [Company] tenant — least-privilege scopes only. SharePoint scope limited to the audit list.

Data

  • Versioned policy ruleset (ruleset_version semver) — every decision row records which version made the call. Mid-cycle bumps re-validate, never silently re-interpret historical rows.
  • SharePoint row schema is semver-versioned because Power BI dashboards depend on it. Breaking changes require a coordinated bump.
  • Audit retention. 7 years on the technical event log; SharePoint row retention per [Company] document-retention policy (assume 7 years until confirmed).
  • GDPR. Recipient and nominator IDs only; no recognition narrative copied into the audit log beyond what’s necessary for review. [People Ops Lead] + [Legal Counsel] sign off the retention model in week 8.

Agent behaviour

  • First failed check halts the cascade. Agent does not re-try or re-weight.
  • Hard monetary ceiling sits outside the cascade — auto-approval refused above it regardless of rule logic.
  • Tool allowlist only. Bounded loops: max 6 tool calls per nomination, 30s wall-clock.
  • Agent never sends to the recipient. Recognition flows through standard Workday R&R notification.

Operational

  • Idempotency. Decisions keyed by nomination_id; a re-fired event must not double-write to the BP or duplicate audit rows.
  • Kill switch. [People Ops Lead] flips one config flag → agent reverts to decision_mode = shadow across the board. Effective in <5 min.
  • Reversibility. Auto-approvals are standard BP completions — reversible by People Ops in one Workday action.
  • Cost ceiling. £150/month Azure OpenAI spend, alert at 80%. Cascade is mostly deterministic — token spend is modest.
10 — Build phases
  • MVP. 3-rule threshold check (value cap, eligibility, frequency) plus exception routing via the existing BP and SharePoint append. One R&R category, one cycle.
  • Shadow mode (cycle 1). decision_mode = “shadow” — rows logged only, no BP writes. Compare agent decisions against human decisions on the same nominations to calibrate before going live.
  • Phase 2. Pattern-detection layer — the agent reads cross-nomination signals (reciprocity, ring detection, gaming patterns) the policy cascade alone can’t see. Multi-category rollout.
  • Phase 3. Generalise the agent pattern (Workday Approval Audit Agent) across other rubber-stamp Workday BPs.

14 — Model choice
  • GPT-4o-mini for the deterministic rule cascade orchestration and tagged-reason formatting.
  • No large model needed — the decision logic is rule-based, not judgment-based. The model is doing orchestration, not reasoning.
  • Hosted on Azure OpenAI on the [Company] enterprise tenant. Zero data egress to public OpenAI.
15 — Riskiest design choice
If we get this wrong, People Ops fixes it in one click The first failed check halts the cascade and routes to a human with a tagged reason — the agent does not weigh, balance, or “argue past” a fail. Conservative thresholds at launch (cap set below the stated policy cap until shadow-mode cycle 1 calibrates against human decisions). Hard monetary ceiling sits outside the rule cascade as an unconditional refuse. Auto-approvals are standard BP completions, reversible by People Ops in one Workday action. Shadow-mode cycle 1 is the hard gate — no auto-approvals go live until agent decisions match human decisions on the in-policy set at ≥95%.

17 — Eval harness

Golden path

Full last cycle (~200 nominations) replayed in shadow mode. Agent decisions logged alongside the human decisions; thresholds calibrated against the delta before live cutover.

Adversarial

  • Value-cap boundary (£1 over, £1 under).
  • Recipient on formal performance process.
  • Frequency boundary — e.g. 4× same recipient/nominator/project in 30 days.
  • Missing recipient record.
  • Reciprocal-nomination pair (A→B, B→A).
  • Ring nomination (A→B, B→C, C→A).
  • Mid-cycle policy version change.

Pre-merge

Every change replays the cycle suite. Tracing on every decision.

PoC passes when…

  • ≥95% agreement on in-policy nominations vs the human baseline in shadow mode.
  • Zero false auto-approves on exceptions.
  • People Ops time per cycle ≤ 30 min.
  • Cycle-to-recognition latency ≤ 24h on auto-approved nominations.

20 — v1 → v2 path

If shadow-mode cycle 1 + live cycle 2 hit the pass criteria — ≥95% agreement, zero false-approves, People Ops time ≤30 min, latency ≤24h — v2 layers in pattern-detection (cross-nomination signals: reciprocity, ring detection, off-policy bursts) and starts generalising the pattern as the Workday Approval Audit Agent across the next rubber-stamp BP.
